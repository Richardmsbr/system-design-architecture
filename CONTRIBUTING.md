# Contributing

Contributions are welcome. This document outlines the guidelines for contributing to this repository.

## Code of Conduct

Be respectful and constructive in all interactions.

## How to Contribute

### Reporting Issues

1. Check existing issues before creating a new one
2. Use the issue template if available
3. Provide clear reproduction steps for bugs
4. Include relevant system information

### Submitting Changes

1. Fork the repository
2. Create a feature branch from `main`
3. Make your changes
4. Ensure documentation is updated
5. Submit a pull request

### Pull Request Guidelines

- Keep changes focused and atomic
- Write clear commit messages
- Update relevant documentation
- Follow existing code style and formatting

## Content Guidelines

### System Design Documents

Each system design document should follow this structure:

```
# System Name

## Overview
Brief description of the system.

## Requirements
### Functional
### Non-Functional

## Capacity Estimation
Back-of-envelope calculations.

## High-Level Design
Architecture diagram and component overview.

## Detailed Design
Deep-dive into each component.

## Database Design
Schema and data model decisions.

## Scalability
Horizontal scaling strategies.

## Failure Scenarios
What can go wrong and how to handle it.

## References
External resources and further reading.
```

### Diagrams

- Use ASCII diagrams for compatibility
- Keep diagrams simple and readable
- Include legends when necessary
- Test rendering on GitHub

### Code Examples

- Use descriptive variable names
- Include comments for complex logic
- Provide context for language/framework choices
- Test code before submitting

## Review Process

1. Automated checks must pass
2. At least one maintainer approval required
3. Address all review comments
4. Squash commits before merge

## Questions

Open an issue for questions about contributing.
